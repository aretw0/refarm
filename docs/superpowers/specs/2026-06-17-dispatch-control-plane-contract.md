# Dispatch Control Plane Contract

This document defines the stack-level contract that lets Refarm route effort/task work across CLIs, daemons, and future runtimes (Telegram, Matrix, HTTP bridges, etc.) without coupling to a specific package.

## 1) Why this exists
Refarm aims to compose multiple packages as blocks. The transport envelope is one of the few layers that must be stable across environments:

- `apps/refarm` (operator CLI)
- `apps/farmhand` (effort execution daemon)
- plugin/runtime adapters
- future hosts (desktop/TUI/browser bots/etc.)

The contract is owned as a shared control-plane surface by:

- `@refarm.dev/dispatch-surface`
- `@refarm.dev/dispatch-surface-rs` (WIT-first canonical substrate, optional runtime fallback)

## 2) Canonical transport values

A transport is one of:

- `file`
- `http`
- `channel:<name>`

`<name>` must be a non-empty string.

Parsing/validation behavior:

- Accepts exact `file`, `http`, and any valid `channel:<name>`.
- Rejects everything else with the exact message:

```text
Invalid task transport "<value>". Use: file, http, channel:<name>
```

This message is intentionally stable and preserved in CLI/parity contracts.

## 3) Channel effort envelope

A channel effort payload is valid when:

- `direction` is a non-empty string.
- `tasks` is an array.

Optional fields are normalized before dispatch:

- `source` defaults to `channel:<name>` when absent.
- `context` merges `channel`, `replyTo`, and `traceIds` when provided.
- `submittedAt` is generated when missing.
- Optional `priority` and `tags` are preserved when present.

Effort builders must produce a canonical object with stable shape for downstream adapters and status tooling.

## 4) Path helpers

Given a base URL and channel, transport clients should use canonical helpers:

- `buildChannelEffortsPath(baseUrl, channel)`
- `buildChannelEffortPath(baseUrl, channel, effortId, segment?)`

Avoid manual URL assembly at call sites; keep this in one place to preserve wire compatibility.

## 5) Channel identifier encoding

Use the paired helpers for transport-safe identifiers:

- `encodeChannel(channel)`
- `decodeChannel(channel)`

The round-trip must be deterministic in both directions.

## 6) Host/daemon integration requirements

A host/daemon that consumes this contract must:

1. Parse transport via `parseTaskTransport(...)`.
2. Route:
   - `file` → file-backed effort transport
   - `http` → HTTP worker transport
   - `channel:<name>` → channel adapter for `name`
3. Build efforts via `buildChannelEffort(...)`.
4. Preserve payload fields with canonical normalization and stable error messages.

### Minimal add-new-channel checklist

To add support for a new channel `channel:<name>`:

1. Register resolver route in host command layer (`apps/refarm`) through existing transport adapter resolution.
2. Add/extend daemon support for status/progress under the same channel namespace (`apps/farmhand`).
3. Normalize effort context consistently (`channel`, `replyTo`, `traceIds`).
4. Add regression tests in both layers to verify:
   - valid/invalid transport parse
   - adapter selection and channel extraction
   - effort payload normalization and route generation

## 7) CI and parity invariants

`@refarm.dev/dispatch-surface` and `dispatch-surface-rs` must remain behaviorally aligned.

- TS parity tests (`test:parity`) assert cross-runtime behavior invariants.
- Root CI should execute parity CI when dispatch transport surfaces change.

## 8) Stability commitments

- Stable public transport string shape: `file`, `http`, `channel:<name>`.
- Stable error message format for invalid transport.
- Stable payload normalization semantics for channel efforts.
- Stable API signatures on `@refarm.dev/dispatch-surface` while Rust/Wasm remains optional.

## 9) Distribution posture

- Native runtime artifacts are optional and do not define behavior.
- `@refarm.dev/dispatch-surface` always has a deterministic TypeScript fallback.
- Optional artifacts (`packages/dispatch-surface-rs/dist`, `packages/dispatch-surface-rs/pkg`) are regenerated by `dispatch-surface:build-rs` (or `dispatch-surface:build-rs:release`, which can hard-fail when jco is unavailable) and treated as build-only artifacts.

## 10) Next-step governance

Any proposed evolution of the control-plane contract should begin as package-local API first; it is promoted to broader package ownership only when the same mechanic is required by at least two independent environments.
