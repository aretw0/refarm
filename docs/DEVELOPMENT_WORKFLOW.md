# The Refarm Development Workflow: SDD → BDD → TDD → DDD

This document formalizes the development process for Refarm core, adapters, and plugins. We follow a "top-bottom" approach, prioritizing architectural decisions and semantic specs before implementation.

---

## 1. SDD: Spec-Driven Development
Everything starts with a schema. Our goal is **Sovereign Interoperability**.

- **WASM Components**: Define the capabilities in WIT (Wasm Interface Type). See [wit/refarm-sdk.wit](../../wit/refarm-sdk.wit).
- **JSON-LD**: Define the data structure for the Sovereign Graph. Every piece of data must have a URI.
- **Contract Interface**: Define the TypeScript interface and the conforming capability (e.g., `storage:v1`).

## 2. BDD: Behavior-Driven Development
Define what "success" looks like before writing a single line of logic.

- **Conformance Suites**: For every contract, we distribute a `run[Contract]Conformance()` helper.
- **Integration Specs**: Describe the interaction between the Tractor and the adapter using human-readable scenarios (e.g., `vitest` unit tests as behavior specs).

## 3. TDD: Test-Driven Development
Iterative implementation guided by the BDD specs.

1. **Red**: Run the conformance test. It fails.
2. **Green**: Implement the minimum logic to make it pass.
3. **Refactor**: Optimize, but stay within the contract gate.

## 4. DDD: Domain-Driven Design
The final layer: cultivating the "Solo Fértil".

- **Sovereign Nodes**: Map domain concepts (Identity, Note, Transaction) to the JSON-LD graph.
- **Tractor Policies**: Orquestrating how different plugins interact with the user's data.
- **Plugin Ingestion**: Normalizing external data into machine-readable sovereign formats.

---

## 5. Tracking Policy: Source vs. Derivatives
To maintain **Grand Style** and avoid repository bloating:

### 1. Grand Style Cleanup
Before large refactors or after switching branches, run:

```bash
npm run clean:derivatives
```

This uses `git clean -fX` to safely purge all ignored build artifacts (`dist`, `*.js`, `*.d.ts`) within `src` directories while protecting tracked source files.

### 2. Dual-Mode Resolution
The project supports two resolution modes to balance high-speed development and regression testing:

- **Source-Only DX (Default)**:
  - Command: `npm run type-check`
  - Purpose: Daily development. Resolves types directly from sibling packages' `src/` directories. Provides instant feedback and robust "Go to Definition" support without requiring manual builds.
- **Dist-Verified CI**:
  - Command: `npm run type-check:dist`
  - Purpose: Pre-release verification and CI. Resolves types from generated `dist/` artifacts. Catches regressions in the build process or missing exports.

- **Track Only Source**: `.ts`, `.wit`, `.ld.json`, `.md`.
- **Ignore Derivatives**: `.js`, `.d.ts.map`, `.js.map` and binary `.wasm` (except specific benchmarks/fixtures).
- **Automation First**: Derivatives are always generated by the CI/CD pipeline or local build scripts. We never commit build artifacts to the repository.

---

## Why this matters?

> "A emergência e convergência tem que vir de casa."

By following this flow, we ensure:
- **Portability**: Data and logic can move between browser and server (Isomorphic Tractor).
- **Security**: Capabilities are strictly gated by the WIT boundary.
- **Speed**: Once the specs (SDD) and behavior (BDD) are set, the implementation (TDD) becomes predictable "cruising speed" work.

---

## Case Study: The Isomorphic Tractor
Our architecture enables **Astro Hybrid** deployment.
- **SSR (Server)**: Tractor uses a `PostgresAdapter` to render for SEO.
- **Hydration (Client)**: Tractor uses an `OPFSSQLiteAdapter` for local-first interaction.
The logic remains identical because the **Contract** stays the same.
