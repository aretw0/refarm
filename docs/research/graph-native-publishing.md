# Sovereign Web & Graph-Native Publishing

**Date**: 2026-03-08
**Status**: Research & Early Specification
**Topic**: How Refarm serves its own interfaces from the Sovereign Graph natively.

---

## 1. The Philosophy: "Graph as Code"

If Refarm is a Personal Operating System for Sovereign Data, its own public interfaces (like the landing page `refarm.dev.br` or a user's blog) should not exist merely as static files in a repository. They should be **nodes in a graph**.

- **The Shell (A Casca)**: The main Refarm binary/app contains only the bare minimum (WASM host, Tractor, and SQLite) and a tiny fallback UI for bootstrap.
- **The Interface as Data**: The entire landing page, CSS, page structure, and routing are synchronized as JSON-LD nodes.
- **The Rendering Engine**: A plugin converts these Graph Data nodes into raw HTTP outputs that browsers (and search engine crawlers) understand.

## 2. Defining the Plugin: `@refarm.dev/plugin-antenna`

Let's name this the **Antenna Plugin** (The Antenna). It acts as an HTTP server, reverse proxy, or static site generator directly coupled to the `Tractor`'s local SQLite store. It broadcasts the Sovereign Graph to the outside world.

### What it Does

1. **Listens to HTTP Routes**: The Tractor forwards incoming HTTP requests (via a Service Worker in the browser or an HTTP port on a Node/Rust server) to this plugin.
2. **Resolves Accessible Nodes (Universal Resolution)**: The plugin queries the Sovereign Graph for public nodes (`@type: WebPage`) matching the requested route.
   - **Crucial Distinction**: These nodes don't have to be natively created in Refarm. Through **Two-Way Data Binding**, a node might represent a Markdown file on GitHub or a row in Google Sheets that another plugin (e.g., a GitHub Integration Plugin) has synced into the Sovereign Graph. The Gateway Plugin doesn't care; it just projects the graph as it is.
3. **Materialization**: If the page node defines dependencies (`script`, `stylesheet`, `image`), the plugin fetches them from the Refarm store (binary blobs) or their bound external sources and renders the final HTTP response.

## 3. Resilience Layers (The 5 Levels)

To ensure this page never goes down in a fragile way (addressing the exact points raised in the brainstorm):

### Level 1: The End of the World (Local Offline)
No servers, no Nostr relays, AWS goes bankrupt.

- **How it works**: You open the Refarm App (running a local Service Worker). The Gateway Plugin intercepts the request, reads your local SQLite, generates the HTML page, and serves it. Refarm continues existing on your machine, completely off-grid.

### Level 2: Pure P2P (Grassroots)
There is at least one peer (a friend) with Refarm online, exposing their port or tunnel.

- **How it works**: Anyone can access `https://purple-rat-23.tunnels.refarm.dev`. That node's Gateway takes the request, reads the "landing page" straight from *their* graph, and serves the visitor. Refarm survives through the grassroots sharing of nodes.

### Level 3: The Anchor Node (refarm.dev.br)
The official page is nothing special. It's just a highly cached node on the network.

- **How it works**: The infrastructure of the official domain runs one or many Tractors. They simply serve the graphs of whoever has the administrative capabilities to sign the "Home Page" node. Deploying a new landing page is, in reality, saving a new *Node* to the official graph and synchronizing it (via CRDT over Nostr).

### Level 4: Edge Delivery Consolidation (The Sovereign CDN)
To prevent a heavy photo from jumping through 15 slow domestic peers via P2P.

- **How it works**: The Gateway Plugin knows which parts of the graph are heavy (video blobs, JS bundles). It orchestrates an external *Storage Tier*. It injects the rendered Graph directly into Cloudflare, AWS CloudFront, or Vercel. The heavy caching is done by massive Edge corporations, but the **Sovereign Truth** (the original content) remains on your local disk in SQLite.

### Level 5: Shared Graph Fragments

- **How it works**: If `refarm.dev.br` goes down, but you accessed it yesterday, your CRDT log (via Nostr/Yjs) might have synchronized a piece of the public page locally and stored it in your SQLite. You can act as a *fallback relay* for isolated public public fragments.

## 4. What Do We Learn From the Ghosts of P2P Past?

By building this plugin, we avoid the traps of older architectures:

1. **The IPFS Mistake (Data Availability vs. Permanence)**:
   - *The Error*: IPFS links die if nobody pays to keep them alive (pinning). Web2 infrastructure cannot read them natively without slow centralized gateways.
   - *The Fix*: Our public data is served via native HTTP (Level 4: Edge CDN) generated by the Gateway but sourced from a SQLite DB (Level 1: Local Offline). You never depend on the kindness of strangers for the landing page to load blazingly fast.

2. **The Secure Scuttlebutt (SSB) Mistake**:
   - *The Error*: The network died if the "append-only log" got too big (onboarding took hours).
   - *The Fix*: The landing page is read from the *Materialized View* in SQLite (as per ADR-028), not by rewinding CRDT logs in the frontend. Time-To-First-Byte (TTFB) is instant.

3. **The Urbit Mistake**:
   - *The Error*: A total rewrite of the OS stack limited the project's reach to its own closed ecosystem (Hoon/Nock).
   - *The Fix*: The shell and the output of the Gateway Plugin use open standards (WASM, JSON-LD, generic HTTP/HTML). The external world consumes Refarm's output without needing to know it was generated by a P2P Sovereign Graph.
