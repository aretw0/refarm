## Track C: Consumer-Pulled Block Supply (`vault-seed`)

`vault-seed` is not merely lab pressure. It is the consumer that prevents Refarm
from polishing abstract blocks while the downstream project keeps rebuilding the
same surface locally. Treat it as v0.1.0 evidence: every useful block should be
proven in Refarm first, then consumed by `vault-seed` through an explicit
candidate channel before any public release promise.

The policy is supplier-first. If an operation in `dgk`, `vault-seed`, or later
`agents-lab` can be powered by a Refarm SDK primitive, engine package, crate,
generator, codemod, or policy contract without harming the downstream audience,
Refarm should supply it and the consumer should import it. Downstream keeps
product UX and vocabulary; Refarm owns reusable machinery.

### Gate C1: Candidate Consumption Channel

- [x] Pack or otherwise expose candidate packages without relying on public npm publication.
- [x] Add a release-policy selection for the packages `vault-seed` can consume as public candidates.
- [x] Record the exact consumer proof command and fallback in the package plan.
- [x] Keep `vault-seed` product UX, copy, routes, notebooks, and PARA conventions downstream-owned.
- [x] Prefer codemods or manifest-driven generation when the consumer change is mechanical.

**Status**: active for item 4 blocks and item 9 generator/codemod work. The build-free HTML helper
surface is now canonical at `@refarm.dev/ds/html` and rides with `@refarm.dev/ds` in
`vault-seed-ready`. `@refarm.dev/homestead-ssr` and `@refarm.dev/homestead/ssr` were removed
pre-publication instead of kept as compatibility surfaces. The full `@refarm.dev/homestead` package stays out of that selection because
publishing the whole package still pulls Tractor/storage/sync/plugin dependencies and SDK-local
alias work. Split or slim that dependency closure before promoting the bundled Homestead SDK.

`process-handoff` follows the same lean rule: `@refarm.dev/process-handoff` is the
publishable leaf in `vault-seed-ready`; `@refarm.dev/cli/process-handoff` remains
a compatibility re-export, while the full CLI package and its `config`/`trust`
closure stay outside the consumer-pulled lane.

As of 2026-06-26, the `vault-seed-ready` publish dry-run passes with this leaf
included and the full CLI closure excluded. The release-readiness plan also runs
`pnpm run codemods:check`, which keeps the codemod registry, fixtures, dry-run
commands, and rollback notes valid before consumer-pulled mechanical adoption.
As of 2026-06-28, it also runs `audience:boundary:test`, keeping SDK primitives
behind explicit audience/naming review before they can move into publication
lanes.
As of 2026-06-29, `pnpm --silent run release:vault-seed:handoff -- --pack
--prune-extra --json --out .refarm/handoff/vault-seed/2026-06-29/manifest.json`
also materializes the tarballs, prunes only unexpected generated `.tgz` files,
writes the collectable `manifest.json`, and emits the acceptance summary. The
operator-readable companion is
`.refarm/handoff/vault-seed/2026-06-29/manifest.md`. After ADR-072 the current
selection is `accepted`, 9 packages, and 20 required checks beside tarball
integrity, so the pre-publication handoff has a machine-readable readiness
signal and rejects stale tarballs or stale publishable build outputs before
reporting `ok: true`. The official `vault-seed` checkout should collect the
same-directory `.tgz` files named by `manifest.json`, not scrape hashes or
package lists from this document.

### Gate C3: Feedback To Daily Driver

Consumer proof feeds Track B instead of waiting behind it:

- a block that `vault-seed` needs must either already be consumed by Refarm itself
  or carry an explicit "consumer-pulled exception" with a Refarm dogfood follow-up;
- if adapting `vault-seed` reveals missing runtime/status/action semantics, the fix
  lands in Refarm's package or host boundary before more downstream edits;
- if the consumer proof requires private checkout knowledge, the proof is not done:
  it must be captured as package docs, fixture, codemod dry-run, or generated
  inventory report.

### Gate C4: Plugin Proof Trigger

The Pi/WASM/UI proof is no longer "after everything." It starts when Track C has
enough block evidence to make the plugin boundary concrete: at minimum `ds`/`ds/html`,
process/artifact handoff, and one credential/provider boundary.
It still gates public `plugin-manifest` release, but it should be planned and
executed as a small consumer-driven proof before plugin-facing APIs are promoted.

### Promotion Matrix: dev → me → social

Use this matrix as the operational rule for all new capabilities.

| Domain                           | Build First  | Promote to `me` when                                                  | Promote to `social` when                            |
| -------------------------------- | ------------ | --------------------------------------------------------------------- | --------------------------------------------------- |
| Storage / Sync / Identity        | `refarm.dev` | Tests green + offline/reconnect validated + no data loss in daily use | Interoperability proven across multiple user setups |
| Tractor runtime                  | `refarm.dev` | 30-day stable use + consumer map validated                            | External operators can run it with docs only        |
| Plugin architecture (Pi/WASM/UI) | `refarm.dev` | 3 plugin types working + inter-layer composition validated            | Third-party plugin authors can publish safely       |
| Automation workflows             | `refarm.dev` | Weekly reliability in your real routine                               | Templates are reusable by others                    |
| UI/UX shell (apps/me)            | `refarm.dev` | Zero-friction daily operation for your workload                       | Community-facing defaults and onboarding are stable |

### Plugin Architecture Proof Window

Plan the Pi/WASM/UI proof now and execute it as soon as Track C has enough block
evidence to make the boundary concrete. It must still run before
`plugin-manifest`, `barn`, `plugin-courier`, or plugin-facing runtime helpers
enter a public release lane.

The proof should stay small and codemod-friendly:

1. one Pi-style agent/skill path that can run through the Refarm host contract;
2. one WASM component path loaded through the manifest/install boundary;
3. one UI surface path rendered through Homestead host-renderer vocabulary;
4. one inter-layer composition where the agent, WASM plugin, and UI surface share
   manifest, trust, action, telemetry, and failure semantics.

Promotion signal: `plugin-manifest` can move from deferred to release candidate
only after this proof has a reproducible command, fixtures, negative trust tests,
and a short authoring example that does not depend on private operator context.

Decision policy:

1. `dev` is the only place for first implementation.
2. `me` is a stability channel, not a prototype channel.
3. `social` only ships patterns that survived real usage in `me`.

## What "v0.1.0" Means Now

- **For You**: Refarm is the daily-driver environment used instead of an external pi instance for planning, coding, memory, automation, and local agent work.
- **For `vault-seed` and adjacent consumers**: Refarm supplies candidate blocks early enough to stop downstream reimplementation, with local/packed/codemod proofs before public release promises.
- **For the Ecosystem**: contracts and reference implementations are publish candidates, not releases, until daily-driver and consumer-pull confidence exist.
- **Future labels**: `v0.2.0+` headings are backlog capability buckets, not automatic version increments.
