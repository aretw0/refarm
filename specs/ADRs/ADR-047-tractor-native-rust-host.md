# ADR-047: tractor-native — Native Rust Plugin Host

**Status**: Accepted
**Date**: 2026-03-19
**Deciders**: Core Team
**Implements**: [ADR-044 — WASM Plugin Loading Browser Strategy](ADR-044-wasm-plugin-loading-browser-strategy.md)
**Related**: [ADR-045 — Loro CRDT Adoption](ADR-045-loro-crdt-adoption.md), [ADR-015 — SQLite Engine Decision](ADR-015-sqlite-engine-decision.md)

---

## Context

`@refarm.dev/tractor` (TypeScript) works correctly in browser and Node.js environments, but carries
the V8 engine as a hard dependency. This creates an adoption barrier for:

- **Edge / IoT**: Raspberry Pi Zero, Android Termux, constrained ARM devices — V8 is too heavy
- **Tauri desktop apps**: embedding a wasmtime runtime is ~10 MB; embedding Node.js is ~80 MB
- **Server-side plugins**: JCO transpilation of `.wasm` components to `.js` adds build complexity
  and removes the determinism guarantee of running the original WASM binary

The existing TypeScript implementation uses:
- `@bytecodealliance/jco` to transpile `.wasm` → `.js` at build time
- `@bytecodealliance/preview2-shim` for WASI P2 stubs
- Both break on non-Node.js environments (no OPFS, no `crypto.subtle`, etc.)

The WIT interface (`wit/refarm-sdk.wit`) is already stable and defines the plugin contract
independently of the host language. A native Rust host can implement the same WIT world without
any transpilation step.

---

## Decision

**Implement a native Rust plugin host (`packages/tractor-native`) using wasmtime.**

The native host:
1. Loads `.wasm` Component Model binaries directly — no JCO transpilation
2. Provides identical WASI P2 + tractor-bridge host functions via `wasmtime-wasi`
3. Uses `rusqlite` (bundled) with the same `PHYSICAL_SCHEMA_V1` schema as the TS implementation
4. Uses `loro` (Rust crate) whose binary format is compatible with `loro-crdt` (JS)
5. Exposes a WebSocket daemon on port 42000, replacing `apps/farmhand` for native deployments
6. Ships as both a Rust library (embeddable) and a standalone binary (~10 MB release)

### Key sub-decisions

#### wasmtime over embedded Node.js / Deno
wasmtime implements the WASM Component Model natively. It reads `.wasm` components produced by
`cargo component` without any intermediate transpilation. Binary footprint in release mode is
substantially smaller than embedding a JS runtime.

#### `rusqlite` with `bundled` feature
Compiles libsqlite3 into the binary. Zero runtime dependency on system SQLite. The schema
(`PHYSICAL_SCHEMA_V1`) is identical to `packages/storage-sqlite/src/index.ts`:

- `nodes(id TEXT PK, type, context, payload, source_plugin, updated_at)`
- `crdt_log(id TEXT PK, node_id, field, value, peer_id, hlc_time, applied_at, FK→nodes)`

A `.db` file written by the TS implementation (browser OPFS or Node.js) can be opened by the
native host and vice versa.

#### `loro` Rust crate (binary-compatible with `loro-crdt` JS)
`loro` (Rust) and `loro-crdt` (JS, v1.10.7) share identical binary update/snapshot formats.
A Loro snapshot exported from a browser session (`doc.export({ mode: 'snapshot' })`) can be
imported by the native daemon (`sync.import_snapshot(bytes)`) without any conversion.

#### `ed25519-dalek` native crypto
Signing is a host primitive, not a plugin feature. Loading `heartwood.wasm` via wasmtime for
every node signature would add WASM→WASM call overhead on every write. `ed25519-dalek` is
linked directly. `SecurityMode` (Strict / Permissive / None) makes crypto opt-out at config level.

#### SecurityMode enforced at `PluginHost::load()` layer
`TrustManager` holds grants. `PluginHost::load()` enforces them when `SecurityMode::Strict` is
active — after SHA-256 hash computation, before wasmtime instantiation. Defense-in-depth: trust
policy cannot be bypassed by constructing the host without checking the manager.

## Architecture

```text
TractorNative (lib.rs)
├── host::PluginHost        — wasmtime Engine + Linker + SecurityMode enforcement
│   ├── RefarmPluginHost    — generated by bindgen! from wit/host/refarm-plugin-host.wit
│   ├── TractorStore        — per-plugin wasmtime Store (WasiCtx + WasiHttpCtx + bindings)
│   └── PluginInstanceHandle — live plugin handle (setup/ingest/teardown/metadata/on-event)
├── storage::NativeStorage  — rusqlite, PHYSICAL_SCHEMA_V1, open_at() for TS .db compat
├── sync::NativeSync        — loro::LoroDoc CQRS + Projector → NativeStorage
│   └── peer_id             — derived from SHA-256(namespace)[0..8] as u64
├── trust::TrustManager     — TrustGrant (plugin_id + wasm_hash + expiry), SecurityMode
├── telemetry::TelemetryBus — tokio broadcast fan-out + RingBuffer<TelemetryEvent>
└── daemon::WsServer        — tokio-tungstenite port 42000, binary Loro update frames
```

## Deployment forms

| Form | Target | Entry point |
|---|---|---|
| `[lib]` | Tauri, CLI, edge agents | `use tractor_native::TractorNative` |
| `[[bin]]` | Standalone daemon | `tractor-native --namespace default --port 42000` |

## Graduation path

When all criteria are met, `tractor-native` becomes the canonical `tractor`:

| # | Criterion | Status |
|---|---|---|
| 1 | All `cargo test` pass | ✅ 49/49 |
| 2 | `BrowserSyncClient` Loro binary interop | ⬜ pending |
| 3 | `simple-wasm-plugin` + `hello-world` load in cargo test | ⬜ pending |
| 4 | TS `.db` readable by `NativeStorage` | ✅ `schema_compat_ts_db_readable` |
| 5 | Binary footprint target | ⬜ open — wasmtime makes ≤15 MB unrealistic |
| 6 | All `@refarm.dev/tractor` consumers mapped | ✅ 4 apps + 8 packages |

Migration steps:
1. `packages/tractor` → `packages/tractor-ts` (preserve as reference)
2. `packages/tractor-native` → `packages/tractor`
3. Update `Cargo.toml name`, npm `package.json name`, path aliases in all consumers
4. Write `ADR-048-tractor-graduation.md`

## Consequences

### Positive
- ~10 MB self-contained binary on edge/IoT (vs ~80 MB with Node.js)
- Same `.wasm` plugins run in browser (JCO) and native (wasmtime) without recompilation
- Same `.db` files are portable between browser sessions and native daemon
- Same binary CRDT format (Loro) enables seamless sync between browser and daemon
- No JCO build step in CI for server-side plugin execution

### Negative / trade-offs
- `cargo component` toolchain required to build WASM plugins for testing (extra dev dependency)
- Two plugin host implementations to maintain until graduation
- Browser interop test (criterion #2) requires a real browser environment — harder to automate in CI
- wasmtime makes 15 MB binary limit unrealistic; criterion #5 needs to be reconsidered

### Neutral
- `BrowserSyncClient` in `packages/sync-loro` requires zero changes — binary WebSocket protocol unchanged
- All WIT contracts unchanged — plugins built once work on both hosts
